01

Windows-First Core Architecture

For maximum power on Windows, do not treat Windows and WSL as competitors. Treat them as separate layers with different jobs. Microsoft positions WSL as the preferred Linux-based development environment on Windows, while Windows-native AI tooling increasingly covers local inference, app integration, and GPU access. [web:27]

Windows Native Layer

PowerShell 7, Windows Terminal, VS Code, Everything, PowerToys, Obsidian, local GUI apps, scheduled tasks, browser automation, file association workflows.

WSL2 Layer

Gemini CLI, Python pipelines, MCP servers, embedding jobs, DuckDB/LanceDB scripts, OCR, shell automation, Docker-hosted services.

Bridge Layer

Use /mnt/c/Users/<you> only for shared user content; keep heavy repos and databases inside the Linux filesystem for performance and inode sanity.

Best-practice split: Windows manages interface, automation launch points, and local model apps; WSL handles package ecosystems, Unix tooling, and most AI pipelines. This gives you stronger tooling, fewer path issues, and better reproducibility. [web:27]

Recommended Directory Map

C:\Users\Shane\Obsidian for notes you edit in GUI apps; ~/ai inside WSL for scripts, environments, models, and databases; C:\Users\Shane\Exports for raw chat/bookmark/PDF drops.

Execution Pattern

PowerShell launches WSL commands, scheduled jobs, and GUI integrations; WSL executes the heavy data transforms; results sync back to Obsidian and SQLite.

Multi-Database Design — Four Layers Instead of One

The strongest Windows personal AI system uses database specialization, not database purity. SQLite is the system-of-record, LanceDB is semantic memory, DuckDB is the analysis engine, and Qdrant is the scale-up server. ChromaDB remains acceptable for prototypes, but it is no longer the best anchor for a serious local-first stack. [web:1][web:4][web:5]

CREATE TABLE sources (
  id INTEGER PRIMARY KEY,
  path TEXT UNIQUE,
  source_type TEXT,
  domain TEXT,
  created_at TEXT,
  modified_at TEXT,
  checksum TEXT
);

CREATE TABLE chunks (
  id INTEGER PRIMARY KEY,
  source_id INTEGER NOT NULL,
  chunk_index INTEGER NOT NULL,
  text TEXT NOT NULL,
  tokens INTEGER,
  summary TEXT,
  FOREIGN KEY(source_id) REFERENCES sources(id)
);

CREATE TABLE extractions (
  id INTEGER PRIMARY KEY,

  extraction_type TEXT NOT NULL,
  value TEXT NOT NULL,
  confidence REAL,
  model TEXT,
  created_at TEXT,
  FOREIGN KEY(chunk_id) REFERENCES chunks(id)
);

Persistent Memory System — Working, Episodic, Semantic, Procedural

Gemini CLI supports persistent context through GEMINI.md files and memory commands, with user and project config scopes in ~/.gemini/settings.json and .gemini/settings.json. [web:6][web:16]

To make it far more powerful, build a four-memory system:

Working Memory

~/.gemini/GEMINI.md — identity, priorities, preferences, current missions, stable constraints.

Episodic Memory

Session summaries written after each major work block: what happened, what changed, what to do next.

Semantic Memory

LanceDB collection of facts, lessons, notes, principles, concepts, and retrieval summaries.

Procedural Memory

Reusable playbooks: install steps, prompts, troubleshooting trees, pipelines, automation recipes.

Critical distinction: chat history is not memory. History is a transcript. Memory is a compressed, queryable representation of enduring utility.
  • Working memory should stay small, explicit, and human-edited.
  • Episodic memory should be appended automatically after sessions.
  • Semantic memory should be embedded and retrievable by meaning.
  • Procedural memory should be stored as versioned Markdown and linked to domains.

File Intelligence Pipelines — Obsidian, PDFs, Chats, Bookmarks, Drafts

To get four times the power, stop treating file analysis as “open file, ask model.” Build multi-stage extraction pipelines: ingest → normalize → chunk → extract → embed → score → route → store.

Ingest Layer

Watch export folders on Windows, then copy into WSL staging folders. Use checksum-based dedupe before processing.

Normalize Layer

Convert everything possible to Markdown or structured JSON. Markdown becomes the universal substrate.

Extraction Layer

Run regex, metadata parsers, and LLM extraction passes to identify principles, URLs, emails, products, gaps, and opportunities.

InputNormalizerFast ExtractorsLLM ExtractorsWhere It Lands
Obsidian MarkdownNone / frontmatter parserg, yq, Python AST-like parsingAxioms, principles, content seeds, missing linksSQLite + LanceDB + backlink note
PDFsDocling to MarkdownRegex, title/heading parser, citation/URL parserSummary, tactics, frameworks, unresolved questionsSQLite + LanceDB
Chat exports JSONjq / Python JSON flattenRole filtering, timestamp extraction, turn groupingDecisions, preferences, recurring themes, reusable promptsSQLite + LanceDB + session summaries
Bookmarks HTMLHTML parse to CSV/JSONDomain extraction, title cleanup, tag inferenceResource clustering, monetization leads, study mapsSQLite + DuckDB
DOCX / draftsPandoc to MarkdownTitle/heading parser, quote and link extractionHooks, offers, threads, scripts, product ideascontent.db + Obsidian
Key upgrade: run two extraction passes, not one. Pass 1 is deterministic and cheap: regex, metadata, headings, dates, entities. Pass 2 is model-based: principles, monetization angles, missing-piece diagnostics, causal structure.

pythonUniversal extraction prompt shape
prompt = f"""
Analyze this document chunk and return strict JSON with:
- axioms: universal truths or durable claims
- principles: reusable heuristics or rules
- urls: any links or cited resources
- emails: any contact addresses
- monetizable_fragments: anything that could become a post, product, service, course, guide, system
- missing_pieces: what the document assumes but does not explain
- action_items: concrete next moves implied by the text
Text:\n{text[:6000]}
"""

Missing-Piece Diagnostics

Ask the model: “What would someone need to implement this in real life that is absent here?” This surfaces hidden dependencies, unstated steps, and missing prerequisites.

Monetization Distillation

Ask the model: “Which fragments here can become paid assets?” Then classify into service, course, ebook, thread, newsletter sequence, worksheet, audit, or toolkit.

Absolute baseline. Gives the model durable file read/write/search capability inside approved paths.

GitHub MCP

Repo search, issue triage, file reading, PR workflows, codebase operations.

Memory MCP

Knowledge graph-style entities and observations; useful for named facts and relationship maps.

Obsidian MCP

Transforms your vault into an active AI-editable knowledge surface.

Model Routing Strategy — Frontier + Local Hybrid

Best overall model for your use case in April 2026: Gemini 2.5 Pro. It is the strongest fit for a Gemini CLI-centered agentic loop because it combines native CLI integration, long context, strong reasoning, strong code performance, and excellent long-document analysis. [web:10][web:13]

Gemini 2.5 Pro

Main synthesis model: deep research, planning, codebase reasoning, large-doc analysis, final architecture decisions.

Gemini 2.5 Flash

High-throughput worker model: extraction loops, batch summaries, classification, corpus triage, cost-sensitive automation.

Local Models via Ollama

Privacy-sensitive or offline tasks, quick drafts, rough coding, local note interaction, and fallback when bandwidth or API budget is constrained.

Best routing pattern: use Flash for ingest-time processing, Pro for deliberation-time reasoning, and a local model for offline or low-stakes supportive tasks.

Recommended Local Models

Llama 4 Scout for giant-context offline analysis; Qwen coder family for local coding and scripting; Gemma 3 for light local note assistance inside Obsidian workflows. Local Obsidian+Ollama setups are increasingly common in 2026. [web:37]

Agent Loop Design

1) Gather with MCP and shell tools. 2) Retrieve semantic memory. 3) Hand working set to Pro. 4) Write outputs to SQLite/Obsidian. 5) Auto-summarize the session into episodic memory.

textOperational model routing policy
- If task = bulk extraction, use Gemini 2.5 Flash
- If task = system architecture, research synthesis, hard debugging, use Gemini 2.5 Pro
- If task = offline/private draft or rough support, use Ollama local model
- If task = indexing pipeline, use local embeddings + deterministic parsers first, model second
Do not let the frontier model do work that grep, SQL, regex, parsers, or deterministic transforms can do more cheaply and more reliably. The model should sit on top of the toolchain, not replace it.
