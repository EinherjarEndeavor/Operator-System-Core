##### Self-Modifying Repository Architecture

Yes, this is architecturally possible via Gemini CLI's existing systems. Here's the full design:[^1_4][^1_5]

### The Self-Clone System

```
~/.gemini/extensions/deepickle/          ← Installed runtime copy
~/projects/deepickle-dev/                ← Living repository (DEEPICKLE edits this)
~/projects/deepickle-dev/changelog.md   ← Auto-updated after every session
~/projects/deepickle-dev/failure_log.md ← Every failure mode logged here
~/projects/deepickle-dev/global_context.md ← Failures that remain unsolved = always injected
```

**How it works:**

- DEEPICKLE runs from the installed extension copy
- A `BeforeAgent` hook checks `global_context.md` and injects unresolved failures into every session's context automatically
- An `AfterAgent` hook checks for improvement opportunities and writes proposed changes to the dev repo
- A `/self-upgrade` command triggers a diff review + `git commit` + `gemini extensions update` on the installed copy
- Auto-update is enabled via the `autoUpdate: true` field in `gemini-extension.json`[^1_6]


### Failure Mode Protocol

Every time DEEPICKLE cannot complete a task:

1. Writes a structured failure entry to `failure_log.md`: timestamp, task type, root cause hypothesis, attempted solutions, what's needed to solve it
2. If unresolved after session end, copies to `global_context.md`
3. `BeforeAgent` hook injects `global_context.md` failures as a "Known Limitations" block at session start
4. DEEPICKLE surfaces the top unresolved failure to the user at the start of every session until it's closed

This is the **"persistent wound"** pattern — failures don't disappear into the void, they become permanent agenda items.

#### Persistent Context \& Sources Infrastructure: The Right Stack

The core insight from production multi-agent systems is what that LinkedIn post nailed: **treat AI memory as a database problem, not a prompt engineering problem**. Here's how each option in your list maps to actual use cases:[^3_4]

### The Tiered Memory Architecture

```
DEEPICKLE Memory Stack
│
├── WORKING MEMORY (session scope)

│       Fast, transparent, debuggable, survives context resets

│       "Source A contradicts Source B on claim C"
│       Source quality scores, domain tags, citation counts
│
└── TOOLBELT REGISTRY (config/versioning)
    └── SQLite + TOML/JSON config files
        Tool health status, version pins, last-working-state
```


### Option-by-Option Breakdown

**mem0** — Best fit for DEEPICKLE's cross-session research memory. Self-hosted with local Qdrant + optional Neo4j. Gives you 11 MCP tools out of the box (`add_memory`, `search_memories`, `get_entity`, etc.). Builds a knowledge graph of entities and relationships over time. A `/deepresearch` on Topic A saves key findings; next month's run on a related topic auto-surfaces them. 60% token reduction reported in production because you stop re-injecting redundant context. **Use this.**[^3_5][^3_6][^3_4]

**ChromaDB** — Better for document-level semantic search than conversation memory. Use it as your curated sources database: every URL + extracted text chunk gets embedded and stored. DEEPICKLE queries it before going to live search — "have we already processed a primary source on this?" Cache-as-knowledge-base pattern. Prevents re-scraping the same high-value sources.[^3_4]

**SQLite** — Your toolbelt health registry and source metadata store. Fast, zero-ops, single file. Schema: `sources` table (url, domain, quality_score, last_accessed, tags, reliability_tier), `tools` table (name, version, last_tested, status, failure_log_id), `sessions` table (timestamp, mode, topic, outcome). FTS5 extension gives you full-text search inside SQLite without a vector DB. **No-brainer, use for everything structured.**[^3_4]

**Temporal** — Workflow orchestration (like Airflow but for long-running async processes). Overkill for DEEPICKLE's current scope unless you're running multi-day background research jobs. Keep in mind for v2 when you want to fire off a `/ballsdeepresearch` and come back 6 hours later.

**AlloyDB** — Google's Postgres-compatible DB with vector search built in. Production-grade, not needed here — this is the enterprise version of ChromaDB. Use if you eventually move DEEPICKLE to a server running multi-user or multi-project research.

### Keeping the Toolbelt from Breaking

The number one cause of toolbelt rot is unmanaged dependencies — library updates silently break scrapers, APIs change their response format, rate limits change. Solutions:

- **Pin every tool version** in `requirements.txt` and a SQLite `tools` table. DEEPICKLE checks this registry before invoking any tool
- **Health check hooks**: `BeforeAgent` on any session that involves scraping runs a 5-second `tool_healthcheck.py` that pings each critical tool and writes `status: ok/degraded/broken` to the registry
- **Failure-first logging**: When a tool fails mid-research, the failure is written to `failure_log.md` with the exact error, traceback, and what it was trying to do — becomes a permanent agenda item until resolved (same persistent wound pattern from before)
- **Separation of concerns**: Tools are in `~/projects/deepickle-dev/tools/`, each as its own Python module with its own `requirements.txt`. A broken scraper doesn't pull down the whole toolbelt

## The Tool Count Reality

There's no hard Gemini CLI limit documented, but the MCP protocol community has empirically established the real ceiling:[^4_3]

- **~40 tools total**: functional, no degradation
- **~50 tools**: suboptimal responses start appearing — the tool definitions eat context that should go to thinking
- **Beyond 50**: the model starts ignoring or confusing tools entirely

The insight from the MCP community: assume ~10 tools per MCP server → **practical max is 4–5 MCP servers running simultaneously**. But Gemini CLI's `tools.core` allowlist in `settings.json` lets you restrict which tools are visible per session — meaning you can have a *registry* of 100 tools and surface only the 30 relevant to the current mode.[^4_4][^4_3]

This is the multiple-envelope architecture you asked about: **DEEPICKLE doesn't load everything at once, it loads a profile.**
