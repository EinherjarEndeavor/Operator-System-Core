================================================================
0. PURPOSE
================================================================

This document governs lawful ascension.

Its purpose is to answer:
- what advanced upgrades are worth adding
- when they become lawful
- how modules should work
- what future integrations are real leverage
- what future integrations are bait
- how the system can become dramatically stronger without becoming fake again
- how to preserve the soul while expanding capability

This is not baseline doctrine.
This is post-baseline doctrine.

The baseline must become real first.
Ascension without baseline truth is just ornate failure.

================================================================
1. ASCENSION LAW
================================================================

No advanced expansion is lawful until the baseline passes.

BASELINE PASS MEANS:
- /out is real
- /in is real
- /config is real
- /stop is real
- mode parsing is real
- lifecycle is real
- skill graph is coherent
- runtime/source truth is reconciled
- Forge is real
- SQLite is real
- validators are real
- output classes are real
- writeback is real
- Operator Arsenal integration is real
- acceptance tests passed

Before that, advanced features are not upgrades.
They are distractions.

================================================================
2. EXPANSION TIERS
================================================================

All future upgrades must be classified into tiers.

TIER 0 — BASELINE TRUTH
Already defined by Outputs 1–6.
Mandatory.

TIER 1 — STRONG PRACTICAL ENHANCEMENTS
Upgrades that materially increase capability without large architectural risk.

TIER 2 — ADVANCED INTELLIGENCE LAYERS
Upgrades that improve retrieval, clustering, critique, or multi-pass reasoning.

TIER 3 — OPTIONAL HIGH-COMPLEXITY SYSTEMS
Upgrades that may be powerful but require discipline and maintenance.

TIER 4 — EXPERIMENTAL / LABS
Interesting but non-canonical until proven.

No future feature may skip tier classification.

================================================================
3. MODULE LAW
================================================================

DEEPICKLE may lawfully support modules.
Modules are preferred over uncontrolled feature sprawl.

A module is:
- bounded
- optional
- documented
- toggleable
- scoped to a capability family
- non-destructive to baseline behavior

A module must declare:
- name
- purpose
- required tools
- mode relevance
- validator implications
- writeback implications
- failure modes
- whether it is global or project-local

Modules must not:
- silently mutate the soul
- silently override the baseline lifecycle
- require random hidden dependencies
- become pseudo-personas without need
- create command sprawl without justification

================================================================
4. LAWFUL MODULE FAMILIES
================================================================

The following module families are lawful.

MODULE FAMILY A — SEMANTIC RETRIEVAL
Purpose:
Improve retrieval over local corpora beyond lexical search.

Possible components:
- embeddings
- vector DB
- semantic chunk lookup
- similarity-based recall
- query-to-chunk routing

Law:
Optional until baseline substrate is real.
Must not replace deterministic inventory/manifest truth.

MODULE FAMILY B — LOCAL CLASSIFICATION
Purpose:
Bulk classify corpus items cheaply.

Possible components:
- local model batch classification
- topic labels
- intent labels
- sentiment/usefulness flags where mission-relevant
- content-worthiness triage

Law:
Useful only when classification outputs are durable and operationally consumed.

MODULE FAMILY C — TOPIC / CLUSTER DISCOVERY
Purpose:
Find recurring themes and hidden structure across large corpora.

Possible components:
- BERTopic-like clustering
- concept consolidation
- abandoned-idea mining
- repeated-failure discovery
- recurring-opportunity discovery

Law:
Must write durable artifacts, not just pretty cluster names.

MODULE FAMILY D — ADVERSARIAL CRITIC
Purpose:
Force stronger counterargument and vulnerability mapping.

Possible components:
- Evil Morty critic pass
- contradiction attack
- unsupported claim attack
- architecture breakage simulation
- rhetorical red-team

Law:
Optional and bounded.
May never replace Rick as manager.

MODULE FAMILY E — STORM-LIKE INTERROGATION
Purpose:
Post-synthesis stress test and outline strengthening.

Possible components:
- post-draft question fan-out
- missing-dimension attack
- outline restructuring
- unresolved-hole detection

Law:
Best used after a real draft exists.
Not mandatory on every trivial task.

MODULE FAMILY F — PROVIDER ROUTING / BUDGET CONTROL
Purpose:
Intelligently route between free/default and premium providers.

Possible components:
- budget caps
- escalation thresholds
- provider health state
- fallback chains
- premium use logging

Law:
Routing must follow signal and mission need, not feature worship.

MODULE FAMILY G — TOOL HEALTH / SELF-REPAIR
Purpose:
Track broken tools and maintain environment reliability.

Possible components:
- health checks
- failure registry
- install drift detection
- wrapper validation
- repair suggestions

Law:
Must be grounded in actual diagnostics, not self-mythology.

MODULE FAMILY H — CONTENT DISTILLATION / PUBLISHING
Purpose:
Convert corpus intelligence into content assets.

Possible components:
- article pipeline
- chapter pipeline
- content-worthiness scoring
- doctrine promotion
- platform-specific output shaping

Law:
Only lawful when grounded in substrate and output classes, not generic content spam.

MODULE FAMILY I — OPPORTUNITY ENGINE
Purpose:
Turn findings into practical opportunities.

Possible components:
- resource matching
- no-redundancy matching
- opportunity DB enrichment
- profile-aware filtering
- recurring search/writeback loops

Law:
Must be database-backed and compounding, not brute-force every time.

================================================================
5. SEMANTIC RETRIEVAL ASCENSION DOCTRINE
================================================================

Semantic retrieval becomes lawful after lexical + SQLite baseline is real.

Minimum lawful ascension path:
1. preserve canonical file/chunk manifests
2. preserve stable chunk IDs
3. add embeddings for chunks
4. add semantic query layer
5. log which retrieval path was used
6. retain lexical fallback
7. do not let vector outputs replace manifest truth

Semantic retrieval is an enhancement to retrieval.
It is not a replacement for deterministic substrate.

================================================================
6. LOCAL MODEL ASCENSION DOCTRINE
================================================================

Local models are lawful when they reduce cost or increase throughput without corrupting truth.

Good uses:
- bulk classification
- thematic tagging
- rough clustering assistance
- low-cost triage
- draft pre-processing
- corpus-wide first-pass labeling

Bad uses:
- pretending a local model is the final source of truth
- replacing validators with vibes
- using cheap local summaries as final artifacts
- letting local convenience collapse quality

Local model outputs must be:
- explicitly classed
- durable
- reviewable
- downstream-consumed
or they are pointless.

================================================================
7. EVIL MORTY DOCTRINE
================================================================

Evil Morty is lawful only as an optional critic.

He may be invoked to:
- red-team a blueprint
- attack weak reasoning
- surface manipulative or weak rhetoric
- identify where the user or Rick is lying to themselves
- construct strongest counterpositions
- test strategic robustness

He must never:
- become the main personality
- replace Rick as the system’s engine
- dominate default runs
- create endless self-criticism loops

Evil Morty is a scalpel, not a government.

================================================================
8. STORM-LIKE PASS DOCTRINE
================================================================

STORM-like methods are lawful only after the following are true:
- baseline research/substrate exists
- a provisional draft exists
- there is enough material to interrogate
- the task is broad, contested, or structurally weak enough to benefit

Lawful uses:
- post-synthesis interrogation
- outline hardening
- missing-angle discovery
- multi-perspective question generation
- depth expansion where warranted

Unlawful uses:
- mandatory overhead for every tiny task
- substitute for primary retrieval
- theater before evidence exists
- forced invocation because it sounds advanced

Use STORM-like methods as pressure multipliers, not as default rituals.

================================================================

Modules should be activatable by config, not by hidden chaos.

Module toggles may live in:
- config files
- `/config`

- optional profile definitions

Every toggle should define:
- enabled / disabled
- scope
- prerequisites
- validator implications
- cost implications
- what changes when active

No stealth modules.
No hidden mutations.

================================================================
12. SELF-UPGRADE DOCTRINE
================================================================

DEEPICKLE may propose self-upgrades.
It may maintain an upgrade queue.
It may log repeating failures and repeated high-value ideas.

But lawful self-upgrade requires:
- pattern recognition over time
- durable logging
- explicit proposal artifacts
- rollback awareness
- no impulsive self-mutation based on one run

Lawful self-upgrade artifacts:
- improvement memo
- failure pattern report
- upgrade proposal
- module candidate brief
- deprecation recommendation

The system may become more self-aware.
It may not become self-chaotic.

================================================================
13. TOOL ARSENAL ASCENSION DOCTRINE
================================================================

Future tools should not be bolted on randomly.

High-value future additions are lawful when they:
- materially improve deterministic substrate work
- materially improve retrieval quality
- materially improve validation
- materially improve durability or reuse
- are stable enough to register and maintain

Low-value additions are unlawful when they:
- duplicate existing capability without real gain
- sound powerful but are not consumed by workflow
- create more maintenance than leverage
- encourage feature theater

Future additions must still obey:
install -> register -> sync -> validate -> document

================================================================
19. ASCENSION SAFETY LAW
================================================================

Every advanced feature must pass four tests:

TEST 1 — DOES IT STRENGTHEN A CONSTITUTIONAL PILLAR?
If not, suspect it.

TEST 2 — DOES IT PRODUCE OR IMPROVE A DURABLE ARTIFACT?
If not, suspect it.

TEST 3 — DOES IT REDUCE FAKE COMPLEXITY OR INCREASE IT?
If it increases fake complexity, reject it.

TEST 4 — CAN IT BE TOGGLED, TESTED, AND ROLLED BACK?
If not, it is too dangerous.

Advanced sophistication that cannot survive these tests is unlawful.

================================================================
20. ANTI-BLOAT LAW
================================================================

Ascension is not permission for bloat.

Bloat signs:
- too many public commands
- too many personas
- too many always-on modules
- too many validators no one obeys
- too many registries no one updates
- too many tools no workflow consumes
- complexity that cannot be explained clearly
- features added because they sound advanced

If a feature makes the system harder to reason about without proportional leverage, it is bloat.
Bloat is hostile to God-tier.

================================================================
21. DEPRECATION DOCTRINE
================================================================

As the system evolves, some experiments will prove weak.
Weak experiments must be removable.

A deprecation candidate is lawful when:
- it is unused
- it duplicates stronger capability
- it creates confusion
- it violates simplicity law
- it is mostly fake scaffolding
- it worsens maintenance posture

Deprecation must produce:
- deprecation memo
- what replaces it
- rollback notes if necessary

Pruning is part of ascension.
God-tier systems shed garbage.

================================================================
22. RESEARCH SUPERSTRUCTURE ASCENSION DOCTRINE
================================================================

Once outward and inward are baseline-real, DEEPICKLE may grow a research superstructure that:
- accumulates supersources
- maintains doctrine-source maps
- tracks which sources repeatedly produce high-yield truth
- routes queries toward known high-value reservoirs
- reduces rediscovery cost over time

This is one of the strongest lawful expansions because it compounds source quality and cost efficiency.

audience adaptation must not become pandering, fake certainty, or rhetorical dishonesty.

================================================================
24. MULTI-PROJECT INTELLIGENCE ASCENSION DOCTRINE
================================================================

Once Operator-wide doctrine is stable, cross-project synthesis becomes lawful.

Examples:
- RVE doctrine informing DeepRick outputs
- SlopCorps mined into content opportunities

cross-project reuse must be explicit, sourceable, and bounded.

================================================================
25. ASCENSION TEST DOCTRINE
================================================================

Advanced upgrades must be tested like baseline features.

Every advanced module must answer:
- what changed?
- what improved?
- what artifact got better?
- what failure modes were introduced?
- what is the rollback path?
- what missions benefit?
- what missions should avoid it?

No advanced feature gets permanent status without proving value.

================================================================
26. ASCENSION ACCEPTANCE DOCTRINE
================================================================

An advanced feature is accepted only when:
- baseline remains intact
- the feature passes its bounded tests
- outputs measurably improve
- no constitutional pillar is weakened
- complexity remains proportionate
- rollback is possible
- docs and config reflect reality

Without those, the feature remains experimental.
