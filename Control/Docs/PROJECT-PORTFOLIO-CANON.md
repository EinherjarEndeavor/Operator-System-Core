# PROJECT PORTFOLIO CANON

Status: Reinforced canon
Purpose: Collapse the major initiatives scattered across Wave1Canon and related piles into a single operational portfolio map with clear hierarchy, governance, and promotion rules.

## 1. Portfolio summary

The current workspace revolves around five primary initiatives:

1. **RVE (Rolling Victory Engine / Revival Vector Engine)**
   - Role: personal operating system for daily execution, constraints, habits, scheduling, and context export.
   - Core value: convert chaos into structured execution.
   - Current bottleneck: truth routing and reliable task-state operation.

2. **Re.Match**
   - Role: constraint-based opportunity matching and dossier generation for justice-involved / recovery / housing-insecure people.
   - Core value: connect a complete profile to live, relevant resources and next-step action plans.
   - Current bottleneck: canonizing service definition, intake fields, matching rules, and dossier production flow.

3. **DeepRick / Mycroft Holmes / Deep Research Engine**
   - Role: research and synthesis engine for turning scattered source piles into high-grade reports and reusable knowledge.
   - Core value: transform fragmented notes and reports into coherent, decision-grade doctrine.
   - Current bottleneck: source curation, canon workflows, and avoiding slop/compression.

4. **Second Wind**
   - Role: broader recovery / reentry / self-transformation endeavor.
   - Core value: translate the user's lived reconstruction into tools, philosophy, and support systems for others.
   - Current bottleneck: clear boundary between mission statement, doctrine, and executable product lines.

5. **Content / Online Endeavors**
   - Role: turn knowledge, systems, and lived experience into portfolio assets, public-facing work, and income paths.
   - Core value: convert accumulated knowledge into outputs people can consume, trust, and pay for.
   - Current bottleneck: scattered positioning, no unified campaign map, too many parallel ideas.

## 2. Portfolio hierarchy

### Tier 1 — Core execution infrastructure
1. RVE
2. Truth routing / command canon / operational control docs
These exist to make every other initiative executable. If these are unstable, all higher-tier initiatives degrade.

### Tier 2 — External service/product initiatives
1. Re.Match
2. Deep Research Engine / Mycroft / DeepRick
These exist to create outward-facing service value and reusable capability.

### Tier 3 — Mission / narrative / leverage layers
1. Second Wind
2. Content / Online Endeavors
These package, frame, and amplify the lower tiers. They do not outrank them.

## 3. Relationship map

### RVE
RVE is the personal execution core. It should support all other projects by maintaining: active tasks, deadlines, anchors, project states, and exportable context.

### Re.Match
Re.Match is the most directly serviceable and externally useful offering. It should draw from: profile truth handling, structured opportunity matching, and dossier generation patterns.

### Deep Research Engine
This is the knowledge refinery. It should feed: RVE doctrine, Re.Match knowledge assets, content pipelines, and operator decision support.

### Second Wind
Second Wind is the umbrella life / mission / transformation narrative. It should contribute: motivational coherence, mission framing, and recovery and reentry doctrine.

### Content / Online Endeavors
This is the monetization and visibility layer. It should package: clean artifacts, explainers, reports, portfolio projects, and public-facing proof of competence.

## 4. Canon boundaries

- **Personal operating system:** RVE, lifestate, task state, command workflows, scheduling.
- **Service platform:** Re.Match intake, matching, dossier generation, opportunity knowledge.
- **Knowledge/refinery system:** Deep Research engine, canonization workflows, report integration, retrieval.
- **Mission / narrative layer:** Second Wind, recovery philosophy, transformation logic, social impact framing.
- **Public-facing / monetization layer:** Website, portfolio assets, applications, consultant materials, educational content.

## 5. Shared-infrastructure rules

### RVE may serve all projects by providing:
- active task storage and project state
- deadlines, anchors, and checkpoint planning
- execution context

### Truth-routing / canon docs may serve all projects by providing:
- source authority rules and routing rules
- acceptance criteria and freeze/defer rules

### No project may silently redefine shared infrastructure:
If Re.Match, Deep Research, or Second Wind require a change to core infrastructure, that change must be explicit and justified.

## 6. What each initiative must produce

### RVE must produce:
- reliable task storage and scoring/ranking
- reliable daily brief / checkpoint outputs
- project-state retrieval

### Re.Match must produce:
- usable intake profile and eligibility-aware matching
- dossier artifact with action steps and non-redundant opportunity recommendations

### Deep Research engine must produce:
- source-grounded reports and canon docs from messy piles
- reusable retrieval assets and decision support for project building

### Second Wind must produce:
- coherent mission statement and doctrine for recovery / self-reconstruction
- social-impact framing that is strong but usable

### Content / Online layer must produce:
- portfolio assets and revenue experiments
- public-facing proof and reusable narrative and educational artifacts

## 7. Project maturity states
Each major initiative must be assigned one of these states:
`IDEA` -> `CANONIZING` -> `MVP_BUILD` -> `MVP_OPERATIONAL` -> `EXTENDING` -> `PAUSED` -> `ARCHIVED`

### Current recommended states:
- RVE — `MVP_BUILD` moving to `MVP_OPERATIONAL`
- Re.Match — `CANONIZING`
- Deep Research Engine — `CANONIZING`
- Second Wind — `CANONIZING`
- Content / Online Endeavors — `CANONIZING`

## 8. Current priority stack & Priority arbitration rules

### Priority 1: RVE operational MVP
Reason: if the personal operating system is not usable, everything else is harder to execute.
### Priority 2: Command/workflow canon + truth routing
Reason: prevents continual re-derivation and canon corruption.
### Priority 3: Re.Match feature canon
Reason: highest external usefulness and strongest path to immediate service value.
### Priority 4: Deep research canonization pipeline
Reason: unlocks the ability to transform the massive corpus into reusable assets.
### Priority 5: Public-facing battle plan
Reason: convert capability into visible artifacts and opportunities.

### Priority arbitration rules:
When two projects compete for time or attention, resolve conflicts in this order:
1. **Operational instability wins first** (If RVE/routing is broken, fix it first).
2. **Near-term irreversible deadlines outrank general improvement work** (Applications/appointments outrank architecture).
3. **Projects with immediate external value outrank speculative future platforms** (Re.Match dossier outranks future-agent design).
4. **Artifact-producing work outranks abstract ideation** (Creating a deliverable beats a brainstorm note).

## 9. Promotion, Pause, and Kill Criteria

### Promotion criteria:
A project may advance from `CANONIZING` to `MVP_BUILD` only when it has:
1. one canon overview
2. one feature-driven spec
3. one clear acceptance test
4. one next-action list inside RVE

### Kill and pause criteria:
Pause or demote a project when one or more of the following are true:
- it has no current artifact target
- it depends on unresolved architecture upstream
- it has become mostly rhetoric rather than operational movement
- it repeatedly steals attention from higher-priority execution systems

## 10. Weekly portfolio review questions
1. Which project produced a real artifact this week?
2. Which project is pretending to be active but is actually rhetorical?
3. Which lower-tier instability is poisoning higher-tier work?
4. Which initiative should be paused for the next cycle?
5. What is the smallest next artifact that proves movement?

## 11. Immediate build principle & What to stop doing

### What to stop doing:
- Treating all files as equal authority.
- Mixing mission prose with operational truth.
- Designing all systems simultaneously.
- Expanding infrastructure before current products have clear operational contracts.

### Immediate build principle:
Each project should now move from scattered brainstorming into:
1. one canon overview
2. one feature-driven spec
3. one current-state / next-state map
4. one active execution list in RVE

## 12. Red-team warnings
The portfolio fails when:
- every project is treated as simultaneously urgent
- mission language substitutes for artifact production
- infrastructure expands faster than daily utility
- public-facing work begins before internal truth is stable
The portfolio succeeds when:
- the core layer stabilizes execution
- one project at a time produces a real artifact
- shared infrastructure is reused instead of reimagined
