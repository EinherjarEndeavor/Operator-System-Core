### SECTION 1: PORTFOLIO SUMMARY & HIERARCHY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The Sovereign Portfolio is organized into three tiers of authority and execution. Stability in lower tiers is a prerequisite for expansion in higher tiers.

#### TIER 1: CORE INFRASTRUCTURE (THE SPINE)
1.  **RVE (Rolling Victory Engine):** Personal OS for daily execution, constraints, and task-state.
2.  **TRUTH ROUTING:** Canonical control docs and routing rules.
*Mandate: If Tier 1 is unstable, all other work is secondary to repair.*

#### TIER 2: SERVICE & CAPABILITY (THE WEAPONS)
1.  **Re.Match:** Constraint-based opportunity matching and dossier engine.
2.  **Deep Research Engine (Mycroft/DeepRick):** Knowledge refinery for turning corpora into doctrine.
*Mandate: Produce externally verifiable service value and reusable capability.*

#### TIER 3: MISSION & AMPLIFICATION (THE CAPE)
1.  **Second Wind:** Broad recovery/transformation collective and doctrine.
2.  **Online Endeavors:** Portfolio assets, public-facing proof, and income paths.
*Mandate: Package and amplify Tier 1/2 results for public consumption and monetization.*

### SECTION 2: RELATIONSHIP & CANON MAP
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- **RVE:** Serves all projects by providing active tasks, deadlines, anchors, and project state.
- **Deep Research:** Feeds RVE doctrine, Re.Match knowledge assets, and content pipelines.
- **Second Wind:** Provides the mission framing and recovery doctrine used by Re.Match.
- **Online Endeavors:** Converts all project outputs into public proof-of-work and revenue.

**CANON BOUNDARIES:**
- **Relational (Tier 2):** SQLite (`lifestate.db`, `rve.db`). SQL-queried exact facts.
- **Semantic (Tier 3):** Neo4j Graph. Relational discovery and pattern matching (Deferred).
- **Synthesized (Tier 4):** Obsidian Vault + `Control/Docs/`. Doctrine and long-form canon.

### SECTION 3: SHARED-INFRASTRUCTURE RULES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1.  **RVE Centrality:** No project may maintain its own independent task list. All execution items MUST reside in `rve.db`.
2.  **Truth Routing:** All projects must follow the `TRUTH-ROUTING-CANON.md` rules for data promotion.
3.  **Explicit Modification:** No project may silently redefine shared DB schemas. Changes must be justified and recorded in `build-state.json`.

**PROJECT STATES:**
`IDEA` -> `CANONIZING` -> `MVP_BUILD` -> `MVP_OPERATIONAL` -> `EXTENDING` -> `PAUSED` -> `ARCHIVED`

**PROMOTION CRITERIA (CANONIZING -> MVP_BUILD):**
1.  One **Canon Overview** (Mission/Purpose).
2.  One **Feature-Driven Spec** (Technical blueprint).
3.  One **Acceptance Test** (Success/Win condition).
4.  One **Active Task List** in `rve.db`.

### SECTION 5: ARBITRATION & GOVERNANCE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

**PRIORITY ARBITRATION (THE LADDER):**
1.  **Operational Instability:** Broken RVE/Routing outranks all other work.
2.  **Hard Deadlines:** Irreversible legal/medical/school dates outrank improvements.
3.  **External Value:** Shipping a Re.Match dossier outranks agentic redesign.
4.  **Artifact over Idea:** Writing a doc beats a brainstorming session.

**WEEKLY PORTFOLIO REVIEW (THE AUDIT):**
1.  Which project produced a real artifact this week?
2.  Which project is pretending to be active but is actually rhetorical?
3.  Which lower-tier instability is poisoning higher-tier work?
4.  What is the smallest next artifact that proves movement?
