# STEP 0 — OPERATOR SUBSTRATE CANON

Status: ACTIVE
Purpose: Define the actual target of Step 0 so implementation does not drift into building a system that serves only RVE.

---

## 1. Executive judgment

Step 0 is **not** “build RVE.”

Step 0 is:
**build a better-than-standard Gemini CLI operating substrate whose main source of compounding power is memory, retrieval, auditability, and self-improvement.**

RVE is an important early consumer of that substrate because it gives immediate daily utility and canonical structured state.
It is **not** the sole design target for the substrate.

If this distinction is lost, the system becomes overfit to one module and fails to support the broader roadmap.

---

## 2. The actual thing being built

The target system is a local-first command substrate for Gemini CLI / Operator that can:

1. preserve structured truth without corruption
2. preserve episodic operational history
3. preserve and retrieve artifacts, docs, and research on demand
4. maintain a machine-readable registry of tools, skills, MCPs, env vars, and capabilities
5. support multiple consumer systems without each one inventing its own memory layer
6. learn from friction, failure, and repeated work
7. expose compact context exports to future agents and memory systems

This means the substrate must serve at least:
- RVE
- Re.Match / Scaffold
- research and deep-research workflows
- website / service workflows
- future learning/tutoring systems
- future communal systems like Second Wind / Forge

---

## 3. Core design doctrine

### Rule 1 — Substrate first
The substrate must be designed as a general operator backbone, not as an RVE-specific shell.

### Rule 2 — Canonical truth before ambient memory
Structured truth must live in explicit canonical stores before vector, graph, or agent memory is allowed to steer outcomes.

### Rule 3 — Rich backend, sparse interaction
The backend may be deep. The visible daily interaction must stay light.

### Rule 4 — Retrieval over pollution
Information should be retrievable on demand rather than injected ambiently into every run.

### Rule 5 — Auditable self-improvement
Any self-improving or self-modifying behavior must be traceable, reviewable, and reversible.

### Rule 6 — Modules consume the substrate
RVE, Re.Match, fitness planning, research, and future systems should consume the same substrate rather than each growing their own separate memory stack.

### Rule 7 — Doctrine and truth remain distinct
Mission statements, values, archetypes, and philosophy belong in doctrine-aware layers, not silently flattened into structured truth.

---

## 4. Required layers

### Layer A — Operator Core
Contains:
- Gemini CLI workspace rules
- GEMINI.md and command conventions
- scripts and wrappers
- extensions / MCP / skills
- tool registry and health status

### Layer B — Canonical Data Layer
Contains:
- structured truth
- structured project/task state
- structured registries
- append-only audit records
- artifact indexes

### Layer C — Memory Layer Family
Contains multiple memory roles:
- structured truth memory
- episodic memory
- artifact memory
- semantic / vector memory
- graph / ontology memory

### Layer D — Self-Improvement Layer
Contains:
- failure mining
- workflow optimization discovery
- skill / prompt / procedure generation
- ontology refinement
- capability-gap detection

### Layer E — Consumer Modules
Contains:
- RVE
- Re.Match / Scaffold
- research engine
- learning / tutoring engine
- website / service surface
- future mission systems

---

## 5. What Step 0 must produce

At minimum, Step 0 should leave behind:

1. a stable workspace layout
2. a stable family of canonical databases
3. a tool / MCP / skill / env registry
4. a session archive and audit trail system
5. an artifact index for docs, notes, and research
6. a clear separation between truth, doctrine, and retrieval layers
7. a context export mechanism for future agents
8. an improvement loop that mines logs and friction

If those things are not present, the substrate is not truly established.

---

## 6. Why RVE still matters early

RVE remains an early priority because it provides:
- immediate daily usefulness
- canonical structured state about obligations, tasks, and life conditions
- a proving ground for the substrate
- a source of context exports that later modules can consume

But the correct relationship is:
**RVE is the first major structured domain on top of the substrate.**
It is not the substrate itself.

---

## 7. Explicit anti-drift warnings

Do not allow Step 0 to drift into:
- building only an RVE app
- making all memory layers ambient by default
- graphing everything before truth routing is stable
- allowing AI memory systems to become the source of truth
- auto-researching every new entry at capture time
- silently mutating prompts, schemas, or skills without audit trails

---

## 8. Final judgment

The Step 0 target is now:

**A local-first Operator substrate for Gemini CLI whose compounding power comes from canonical truth, layered memory, auditability, retrieval, and self-improvement — with RVE as the first major consumer, not the whole system.**
