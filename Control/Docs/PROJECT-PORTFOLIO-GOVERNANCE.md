# PROJECT PORTFOLIO GOVERNANCE

Status: Proposed hardening addendum
Parent document: `PROJECT-PORTFOLIO-CANON.md`
Purpose: Turn the portfolio map into an operating governance contract so projects stop competing chaotically for attention, infrastructure, and authority.

## 1. Portfolio hierarchy

### Tier 1 — Core execution infrastructure
1. RVE
2. Truth routing / command canon / operational control docs

These exist to make every other initiative executable.
If these are unstable, all higher-tier initiatives degrade.

### Tier 2 — External service/product initiatives
1. Re.Match
2. Deep Research Engine / Mycroft / DeepRick

These exist to create outward-facing service value and reusable capability.

### Tier 3 — Mission / narrative / leverage layers
1. Second Wind
2. Content / Online Endeavors

These package, frame, and amplify the lower tiers.
They do not outrank them.

## 2. Priority arbitration rules

When two projects compete for time or attention, resolve conflicts in this order:

1. **Operational instability wins first**
   - If RVE, truth routing, or command canon are broken, fix them before expanding public-facing systems.

2. **Near-term irreversible deadlines outrank general improvement work**
   - Applications, appointments, obligations, and time-sensitive opportunities outrank research and architecture improvement.

3. **Projects with immediate external value outrank speculative future platforms**
   - Re.Match dossier production outranks broad future-agent system design.

4. **Artifact-producing work outranks abstract ideation**
   - Creating a deliverable canon doc beats another brainstorm note.

## 3. Shared-infrastructure rules

### RVE may serve all projects by providing
- active task storage
- project state
- deadlines and anchors
- checkpoint planning
- execution context

### Truth-routing / canon docs may serve all projects by providing
- source authority rules
- routing rules
- acceptance criteria
- freeze/defer rules

### No project may silently redefine shared infrastructure
If Re.Match, Deep Research, or Second Wind require a change to core infrastructure, that change must be explicit and justified.

## 4. Project maturity states

Each major initiative must be assigned one of these states:
- `IDEA`
- `CANONIZING`
- `MVP_BUILD`
- `MVP_OPERATIONAL`
- `EXTENDING`
- `PAUSED`
- `ARCHIVED`

### Current recommended states
- RVE — `MVP_BUILD` moving to `MVP_OPERATIONAL`
- Re.Match — `CANONIZING`
- Deep Research Engine — `CANONIZING`
- Second Wind — `CANONIZING`
- Content / Online Endeavors — `CANONIZING`

## 5. Kill / pause criteria

Pause or demote a project when one or more of the following are true:
- it has no current artifact target
- it depends on unresolved architecture upstream
- it has become mostly rhetoric rather than operational movement
- it repeatedly steals attention from higher-priority execution systems

## 6. Promotion criteria

A project may advance from `CANONIZING` to `MVP_BUILD` only when it has:
1. one canon overview
2. one feature-driven spec
3. one clear acceptance test
4. one next-action list inside RVE

## 7. Weekly portfolio review questions

Use these questions during review:
1. Which project produced a real artifact this week?
2. Which project is pretending to be active but is actually rhetorical?
3. Which lower-tier instability is poisoning higher-tier work?
4. Which initiative should be paused for the next cycle?
5. What is the smallest next artifact that proves movement?

## 8. Red-team warnings

The portfolio fails when:
- every project is treated as simultaneously urgent
- mission language substitutes for artifact production
- infrastructure expands faster than daily utility
- public-facing work begins before internal truth is stable

The portfolio succeeds when:
- the core layer stabilizes execution
- one project at a time produces a real artifact
- shared infrastructure is reused instead of reimagined
