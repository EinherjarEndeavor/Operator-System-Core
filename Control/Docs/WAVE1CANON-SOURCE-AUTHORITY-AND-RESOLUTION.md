# WAVE1CANON SOURCE AUTHORITY AND RESOLUTION

Status: Proposed hardening addendum
Parent document: `WAVE1CANON-CANONIZATION-PIPELINE.md`
Purpose: Prevent repeated canon corruption by defining source classes, authority order, and duplicate/conflict handling.

## 1. Source authority classes

### Class 0 — Direct live clarification
Definition: direct user statement in current or later operating session.
Authority: highest.

### Class 1 — Structured profile / answer sources
Definition: fielded or semi-structured source files intended to capture facts and constraints.
Examples:
- structured answer files
- onboarding answers
- direct intake exports

### Class 2 — Current accepted canon docs
Definition: current control docs accepted as operational references.
Examples:
- truth routing canon
- feature-driven canon specs
- hardened contracts

### Class 3 — Strong design/source doctrine
Definition: stronger build or design instructions that define intended system behavior.
Examples:
- RVE master build instruction docs
- memory system proposition docs

### Class 4 — Artifact and narrative sources
Definition: persuasive, presentational, or voice-driven documents.
Examples:
- cover letters
- resume summaries
- mission statements

### Class 5 — Raw duplication / passover / brainstorm piles
Definition: repeated, partial, duplicated, or lower-authority exploratory material.
Authority: lowest unless promoted through canonization.

## 2. Resolution rules

### Rule A — Higher class outranks lower class
If a Class 0 or Class 1 source conflicts with a Class 4 or Class 5 source, the lower class does not silently survive as equal truth.

### Rule B — Current accepted canon outranks old doctrine
If a current accepted control doc intentionally narrows or replaces an older design idea, the older idea becomes background, not active doctrine.

### Rule C — Narrative is not fact by default
A statement from an artifact or pitch doc must be extracted and justified before it becomes structured truth.

### Rule D — Duplicates do not create consensus
Five duplicated passover files saying the same thing do not count as five independent confirmations.

## 3. Duplicate handling

When multiple files repeat the same concept:
1. identify the highest-authority version
2. route the rest to supporting/deferred/archive status
3. do not re-canonize the same concept from each duplicate source

## 4. Conflict handling

When sources conflict:
1. preserve the conflict note
2. identify source classes involved
3. prefer the higher-authority source
4. if conflict remains material, require explicit review

## 5. Unresolved ambiguity policy

Mark unresolved ambiguity when:
- two same-class sources conflict materially
- a fact is important but only weakly evidenced
- a narrative claim appears useful but not stable enough to canonize

## 6. Current practical authority ladder

For current-phase operator work:
1. direct current clarification
2. structured truth sources
3. accepted control/canon docs
4. stronger design doctrine
5. artifact prose
6. raw passover and duplicate piles

## 7. Archive rule

A source should be treated as archive/deferred when:
- it adds no distinct operational value
- it mostly repeats stronger sources
- it is useful as historical ore but not as current authority

## 8. Success condition

The operator should not have to mentally remember which of five similar files is the real one.
The canon and authority ladder should make that obvious.
